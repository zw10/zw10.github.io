Betting on ball part I: Getting some data

7 June 2018 in data science

Right, so the world cup is around the corner - what better way to celebrate than to see if we can make some money from gambling? Before anything else, I’ll need to go hunting for some data.

Where to get the data?

First of all, if you just want the historical world cup match scores you can find them here. But if you want to build your own relational database of football data so that you can build a better predictive model, then read on.

So thanks to Gerald Bauer for the data - please do reach out to him if you want to contribute - I’m also trying to do my part with this post, which should hopefully make it easier to get started with sportdb.

Because the data was in the mini football data language, I debated for a while whether to use sportdb, the command-line utility provided or just write my own parser to access the flat files directly - which would also avoid having to navigate any database connections.

Not wanting to reinvent the wheel, I opted to use sportdb. Note that there is an easier way to access the data using JSON (which was created almost at exactly the same time as this post), but there’s much less data available.

The installation process for the sportdb was considerably more involved than I had bargained for, because I was lacking a bunch of prerequisites, so maybe I should have just written that parser instead. Oh well.

The documentation was fragmented across different repositories, so it actually took two afternoons to figure out how to get at the data. Again assuming that you’re running Ubuntu, just paste the commands below in your terminal to get at the data - it might take a while to finish running.

Poking around

In above screenshot I’m exploring the database through a browser with sportweb. If you want to do likewise, just input the following into your terminal.

Navigate to localhost:3000 to check out what’s actually in the database. There is a guide to the schema that the database follows, and a nice diagram that illustrates it.

Once you’ve poked around to your heart’s content, to continue working with the data, you’ve two options:

  1. Use the sqlite3 command-line interface (CLI) to query the table directly - I didn’t realise that running SQL queries directly from the terminal was a ‘thing’ before attempting this project either, but it actually feels natural enough. Too pro by half?
  2. Use a tool like SQLiteStudio, which is more graphical and recommended if you’re working with more complex schema (as is the case here). More my style

Scroll down and and go through whichever section is relevant to you.

Sqlite3 CLI

Navigate to the folder with the database file, and type sqlite3. You should see something like the below. Type in .open sport.db, and .tables to see the list of available tables. Try running some queries to get used to the interface, and don’t forget the semi-colons at the end of each line! When you’re done, type .quit to terminate.

SQLiteStudio

Go to Database -> Add a database. Find your database folder, and the database file itself, named sport.db by default. In ‘Name (on the list)’, type in the name you’d like the database to show up as in SQLiteStudio - don’t worry, you can change this later by right-clicking on the name and clicking ‘Edit the database’.
